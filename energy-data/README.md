# Energy Data

This folder contains the reference data and scripts used to build the energy traces consumed by the experiments.

## Files

1. `build_energy_traces.py` is the single pipeline entrypoint. It can call the ENTSO-E API, export Batsim-compatible traces, or do both.
3. `energy_mix.toml` is the shared configuration for both collection and export.
5. `intensities.json` is the lookup table of carbon and water intensity factors per energy technology.
6. `traces/*.csv` are generated Batsim traces consumed by `--environmental-footprint-dynamic`.

## Pipeline

The workflow still has two stages internally:

1. Fetch one ISO week from the ENTSO-E API into a local CSV snapshot.
2. Re-run the export stage offline as many times as needed with different source-to-intensity mappings or output selections.

Both stages are controlled by one script and one TOML config. If the config is missing or malformed, the script exits with an error and does nothing.

## Run

Run the pipeline inside the Nix shell:

```bash
nix develop --command python3 experiments/energy-data/build_energy_traces.py --config experiments/energy-data/energy_mix.toml
```

The config controls:

- `fetch` to enable or disable the ENTSO-E API call
- `export` to enable or disable trace export
- the ISO `year` and `week`
- an optional `output_csv` for the normalized intermediate CSV
- the list of extracts, each with a country ISO code and an optional dataset name
- an optional `intensities_json`
- an optional list of outputs
- an optional `source_mappings` override table

If `fetch = true`, the script calls the ENTSO-E Web API, hardcodes a 30-second request timeout, resolves country EIC codes from ENTSO-E's official approved EIC XML list, and requires an API token in `ENTSOE_API_TOKEN` or `API_TOKEN`.

If `export = true`, the script reads the normalized CSV and writes one `traces/<dataset_name>_trace.csv` file per dataset unless explicit outputs are configured.

If `fetch = true` and `export = true`, the export stage uses the CSV generated by the same run.

If `fetch = false` and `export = true`, the script expects the normalized CSV to already exist at `output_csv`, or at the default `entsoe-data/entsoe_<year>_W<week>.csv` path when `output_csv` is omitted.

## Normalized CSV

The normalized CSV stores one row per source and time interval, with enough metadata to reproduce the export step offline:

- `dataset_name`
- `country_code`
- `country_eic`
- `source_code`
- `source_name`
- `interval_start_utc`
- `interval_end_utc`
- `generation_mw`

It also stores the requested time window and period metadata for traceability.

If an extract omits `name`, the dataset name defaults to the country ISO code.

The ISO week boundaries are computed per extract in the country's local timezone, Monday 00:00 through the following Monday 00:00, then converted to UTC for the API request.

If `output_csv` is omitted, the pipeline uses `entsoe-data/entsoe_<year>_W<week>.csv`.

## Export stage

The export step fails hard when:

- the input CSV is missing required columns
- a configured dataset does not exist
- an active ENTSO-E source has no mapping
- a mapping references a missing key in `intensities.json`
- a dataset has malformed rows

Country-level ENTSO-E responses can expose multiple parallel series for the same source and interval. The export stage sums those rows before computing the energy mix percentages.

The generated trace format now uses a synthetic single-source mix:

- `timestamp`
- `host_id`
- `property_name`
- `new_value`

The export stage writes a single static row at `t=0`:

- `energy_mix = Mix:100.00`

Then it writes one `carbon_intensity` row and one `water_intensity` row per interval, using the `resolution_seconds` values from the normalized ENTSO-E data. If a dataset mixes resolutions, the export uses the smallest interval and repeats coarser source values to match it.

Each effective intensity is computed from the weighted source shares in that interval:

- `W_j = E_j / sum(E_k)`
- `I_effective = sum(W_j * I_j)`

Negative generation rows are clamped to zero and logged as warnings. Intervals with zero or negative total generation are exported with `Mix:0.00` for both intensities and logged as warnings.

Timestamps are aligned to Monday 00:00 of the requested ISO week in each country's local timezone. That local Monday becomes `timestamp = 0`.

We only update `master_host` through a single host id, currently `AS0`. It must still correspond to the first host in the platform file.

## Default datasets

The default TOML config targets the same three winter-week datasets used by the experiments:

| Country | Dataset | ISO code | Role |
| :--- | :--- | :--- | :--- |
| France | `clean_energy` | `FR` | low-carbon scenario |
| Poland | `fossil_heavy` | `PL` | fossil-heavy scenario |
| Germany | `mixed` | `DE` | mixed scenario |

The default config uses ISO week 2 of 2026. Each dataset is requested and exported in its own country-local Monday-to-Monday window.

## Intensity Factors (`intensities.json`)

`intensities.json` is the lookup table of intensity factors for water consumption and carbon emissions from energy generation technologies. The water factors are from Macknick et al. 2012, while carbon factors are from IPCC 2014 and UNECE 2020 reports. The units are L/kWh for water and gCO2eq/kWh for carbon. Water values were converted from gallons/MWh to L/kWh.

Use composite keys to look up technologies:

- Water: `fuel-technology-subtype-cooling-ccs`
- Carbon: `fuel-technology-subtype-cooling-ccs-source`

Use defaults when technology details are unknown:

- Water: `coal-default`, `gas-default`, `nuclear-default`, `csp-default`, `biopower-default`, `geothermal-default`, `hydro-default`, `solar-pv-default`, `wind-default`
- Carbon: `coal-default-ipcc-2014`, `coal-default-unece-2020`, `gas-default-unece-2020`, `solar-pv-default-unece-2020`, and related keys

The pipeline has a built-in mapping for the fixed ENTSO-E source list. By default it resolves each source to the generic labels in `intensities.json`, prefers UNECE 2020 default carbon keys when they exist, and falls back to IPCC 2014 only when no UNECE default exists. The `source_mappings` table is optional and only needed to override those defaults.
